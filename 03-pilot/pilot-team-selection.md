# Pilot Team Selection

## Selection Criteria

Choose 2–3 teams that maximize learning while minimizing risk:

| Criteria | Weight | Notes |
|----------|:------:|-------|
| Team enthusiasm / willingness | High | Champions drive adoption |
| Language diversity (cover top languages) | Medium | Validates Copilot across stack |
| Mix of greenfield + legacy code | Medium | Tests both creation and maintenance scenarios |
| Low regulatory risk | High | Avoid PCI/HIPAA repos in pilot |
| Measurable output (PRs, velocity) | Medium | Need baseline metrics |
| Team size (5–15 ideal) | Low | Big enough for data, small enough to support |

## Pilot Duration

| Milestone | Target Date | Criteria to Proceed |
|-----------|:-----------:|---------------------|
| Pilot start | | Seats assigned, training complete |
| Week 2 check-in | | >60% active usage, initial feedback |
| Week 4 review | | Metrics baseline vs. current, satisfaction survey |
| Pilot end / go/no-go | | Decision to expand based on success metrics |

## Risk Mitigation

| Risk | Mitigation |
|------|-----------|
| Low adoption in pilot | Weekly check-ins, pair programming sessions with Copilot |
| Code quality concerns | Code review process unchanged, monitor PR review feedback |
| Security incident from suggestion | Content exclusions + existing code review + secret scanning |
| Developer pushback | Voluntary opt-in during pilot, emphasize augmentation not replacement |
